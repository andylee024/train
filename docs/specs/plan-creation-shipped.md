## Phase 1 В· Intake

**Goal:** Extract minimum signal needed to personalize the marketplace.

**Captures:**
- Goals (multi-select chips, в‰Ґ1 required)
- Days/week (single-select chips, required)
- Constraints (multi-select chips, optional)

**Component:** `components/plan/goal-intake.tsx`

**State:** `lib/use-intake.ts` вЂ” localStorage-backed (`plan.intake.v1`)

**Goal chips (9):**
`stronger`, `build-muscle`, `jump-higher`, `run-faster`, `look-ripped`, `more-flexible`, `hybrid`, `longevity`, `sport-prep`

**Constraint chips (9):**
`shoulder`, `knees`, `wrists`, `lower-back`, `hips`, `limited-equipment`, `no-gym`, `time-constrained`, `travel-often`

**Returning user behavior:** If `plan.intake.v1` exists in localStorage on mount, page initializes with `phase = "marketplace"`, skipping intake.

## Phase 2 В· Marketplace

**Goal:** Athlete picks 1-4 coaches whose programs will be synthesized into a plan.

**Component:** `MarketplacePhase` inside `app/plan/new/page.tsx`

**Composition:**
- Hero with `Matched for: <goals> В· <days/wk>  [edit]` line (edit returns to intake)
- `<RecommendedBand>` вЂ” top 3 coaches by goal match (`lib/matching.ts:topMatches`)
- `<FilterBar>` вЂ” search + category chips + goal/level dropdowns
- Coach grid (sorted by match score)
- `<SelectionBar>` вЂ” sticky bottom; pills + "Build plan в†’"

**Data:**
- `lib/coaches.ts` вЂ” 12-coach roster (id, name, handle, category, tagline, stats, tags, philosophy, bestFor, notFor, pairsWith)
- `lib/coach-profiles.ts` вЂ” extras for profile route (principles, weekStructure, videos)
- `lib/matching.ts` вЂ” `scoreCoach(coach, goals)`, `topMatches(coaches, goals, n)`, `GOAL_TO_COACH_TAGS` mapping

**Categories (4):**
`strength` (blue), `athletic` (orange), `aesthetic` (pink), `hybrid` (green) вЂ” accents in `CATEGORIES` const.

**Selection state:** `lib/use-selection.ts` вЂ” localStorage-backed (`plan.selection.v1`), shared with profile route.

**Click-throughs:**
- Card body в†’ `/plan/coaches/[id]` (profile)
- Card `+` button в†’ toggles cart (no nav)
- Cart pill `Г—` в†’ remove
- Cart "Build plan в†’" в†’ review phase

**Profile route:** `app/plan/coaches/[id]/page.tsx` вЂ” full hero, philosophy, principles, sample week, sample content, fit (best for/not for), pairs-well-with chips. Selection bar persists at bottom.

## Phase 3 В· Review (blend)

**Goal:** Athlete confirms coach picks + sees day-by-day shape before committing to synthesis.

**Component:** `components/plan/review-blend.tsx`

**Composition:**
- "Your training team" вЂ” list of picked coaches with category + first 2 goal tags
- "Your week shape" вЂ” 7-day grid mock-generated by `lib/blend.ts:generateBlend`
- "Adjust" вЂ” inline days/wk picker (updates intake live)
- "Anything else to know?" вЂ” optional textarea (`reviewNotes` local state, will be passed to synthesizer)

**Blend generation (v1 mock):** `lib/blend.ts:generateBlend(coaches, daysPerWeek)`
- Distributes coaches round-robin across training days
- Picks session labels by coach category (`SESSION_BY_CATEGORY` map)
- Rest/recovery on days past `daysPerWeek`
- Deterministic вЂ” same inputs в†’ same week

**Click-throughs:**
- "change my picks" в†’ back to marketplace (cart intact)
- "Build plan в†’" в†’ synthesizing phase

## Cross-cutting concerns

### Selection bar persistence

`components/plan/selection-bar.tsx` renders on three surfaces with the same data source (`useSelection`):
1. Marketplace вЂ” pills + Clear + Build plan
2. Coach profile вЂ” pills + Clear + Build plan (Build в†’ `/plan/new?build=true`)
3. (Not on review/preview вЂ” those have their own actions)

URL trigger `?build=true` on `/plan/new` bypasses marketplace and jumps straight to review phase.

### LocalStorage keys

| Key | Purpose | Hook |
|---|---|---|
| `plan.intake.v1` | Goals, days/wk, constraints | `useIntake` |
| `plan.selection.v1` | Selected coach IDs | `useSelection` |
| `dashboard.v2.*` | Widget engine layout overrides | `useDashboardConfig` |

To reset the plan-creation flow during testing: clear `plan.intake.v1` and `plan.selection.v1` in DevTools.

### Filtering vs ranking

`FilterBar` filters (hard cut). Goal matching ranks (soft sort). Both apply to the same grid:
1. `FilterBar` filters out non-matching category/level/search.
2. If `intake.goals.length > 0`, the surviving rows are sorted by `scoreCoach` desc.
3. Recommended band is the top-3 by score (across all coaches, ignoring filter bar).
